Detailed Description

Object Oriented Programming with C.

Declaring a class

Basic principles

Declaring a new class needs quite some boilerplate code. First you have to define two macros, named ${MY_CLASS}_FIELDS(pfx) and ${MY_CLASS}_METHODS(type_t). Both those macros are functional.

Those two macros must call the macros associated to the class you are subclassing. Then you can add the new members your class have into the _FIELDS macros, and its methods and static members into the _METHODS one.

Every class must at least subclass object. It's not possible to create new base types.

Then, to really define your class, you must call the macro OBJ_CLASS this way:

OBJ_CLASS(${my_class}, ${my_super_class}, ${MY_CLASS}_FIELDS, ${MY_CLASS}_METHODS);

For example, if we had the class shape and that you want to subclass it into circle you could for example write this code:

#define CIRCLE_FIELDS(pfx) \
SHAPE_FIELDS(pfx); \
point_t center; \
int radius
#define CIRCLE_METHODS(type_t) \
SHAPE_FIELDS(type_t); \
void (*draw)(type_t *self)
OBJ_CLASS(circle, CIRCLE_FIELDS, CIRCLE_METHODS, shape);

Module counterparts

Once this is defined, you have to put in one single and only C module the virtual table initialization. This is achieved using the OBJ_VTABLE{,END} macros, like in the example:

OBJ_VTABLE(circle)
circle.area = &circle_area; // override a shape method
circle.draw = &circle_draw; // define the implementation
OBJ_VTABLE_END(circle)

Note about initializers and wipers

Each class has a init and wipe method. Those, unlike other virtual methods are meant to only initialize/deinitialize the members that are added in this subclassing step. It means that you should not call the super class implementations by yourself.

The object machinery ensures that all initializers of a class hierarchy are called from the lower level to the highest level ones (with respect to the subclassing ordering). The same is true for the wipe methods, you only need to deallocate the fields that are specific to this class. wipe functions are called in the reverse order of the init ones.

Note that objects are always memset to zero, so it's only necessary to initialize fields that need a non zero initial value.

Objects should not provide a generic new or delete method. Though it's possible to implement various factories that will generate objects properly.

Objects are refcounted in builtin, with two methods: retain and release working like in ObjC. Objects are either created as automatic variable with obj_init and released with obj_wipe, or on the heap using obj_new and obj_release (or obj_delete which is an alias). Using obj_{retain,release,delete on an object created using obj_init is invalid.

Dive into the hacks

Don't mind the argument to the _FIELDS macros, it serves an obscure matter too horrible to discuss. The argument to the _METHODS macro though is very interesting and important. When you use it in the methods return types or arguments types, it will be expanded to the object type you are defining.

This way, you can emulate C++ this easily.

Another cool hack, is that despite the _METHODS macro clearly being meant to define only callbacks, it's possible to put mere members in it. Those will become the equivalent of static class members in C++. Though unlike C++, those fields will be virtual, which can lead to surprising results.

Everyday's objects use

OBJ_CLASS tools

For each class you define, the OBJ_CLASS macros gives you many things. For example, for a class named frob, you get the following:

  • an object and a vtable structure type: frob_t (the object type) and a frob_class_t.
  • a function named frob_class() that will return to you the sole frob_class_t instance that is in fact your class virtual table. This function implementation is generated thanks to the OBJ_VTABLE{,END} macros we described above.
  • a function named frob_super() that returns to you your super class virtual table. This is not really meant to be used by programmers, and is actually only leveraged by the OBJ_VTABLE macro, but some may find it other uses.

Instantiating objects

Intersec Objects are the closest possible to plain C structures. This means that you can either use them on the stack, or allocate them. Though, for stack allocated objects, remember that it's not C++, there is nothing like automatic objects.

For a frotz_t type, it's hence possible to do any of the following:

// object allocated on the stack
{
frotz_t frotz;
obj_init(&frotz);
// ...
obj_wipe(&frotz);
}
// allocated object
{
frotz_t *frotzp;
frotzp = obj_new(frotz); // `frotz_t *frotzp = new frotz();`
// ...
obj_delete(&frotzp); // `delete frotzp;`
}

Create a tagged reference on the object.

A tagged reference has to be released using obj_tagged_release, with the same tag as the one it was originally retained with. Of course, a tagged reference will preferably be released in the same file as the one in which it is created.

Tagged references allow to track actively how the refcounting is used for an object. It has an informative value: it makes the correspondance between retain and release obvious.

Possible evolutions:

  • forbid using non-tagged reference for some objects with a dedicated version of obj_new() or for a whole class with a class attribute.
  • trace mode: to trace all the retain/release done during an object's lifetime.
