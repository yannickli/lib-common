27/* IOP² - An IOP-based library for IOP generation.
28 *
29 * Tools for dynamic generation of usable IOP content.
30 *
31 * iopsq.Package ── 1 ──→ iopc_pkg_t ── 2,3 ──→ iop_pkg_t
32 *
33 * 1. Building of an iopc package from an IOP² package. The code is in
34 * iopc-iopsq.c and it works in a similar way as iopc-parser.
35 * The package can refer to types from IOP environment: builtin IOP types
36 * can be used and have the expected pointer value (&foo__bar__s).
37 *
38 * 2. Resolution of IOP types with iopc-typer.
39 *
40 * 3. Generation of an IOP C package description. The code is in iopc-lang-c.c
41 * because it does quite the same as the code generator except that it
42 * directly generates the structures. This is the part that uses the
43 * provided memory pool.
44 *
45 * Limitations:
46 *
47 * - Default values: default values exist in iopsq.iop and are correctly
48 * transformed at step 1., but not at step 3 (yet).
49 *
50 * - Not supported yet: classes, attributes, modules, interfaces, typedefs,
51 * RPCs, SNMP objects.
52 *
53 * - Typedefs from IOPs loaded to the environment cannot be used by
54 * refering to them with "typeName" like for the other types because
55 * typedefs cease to exist in packge descriptions iop_pkg_t, so they are
56 * missing in the IOP environment too.
57 *
58 * - Dynamic transformation of IOP syntax (mypackage.iop) into IOP
59 * description without having to create a DSO: already possible but no
60 * helper is provided to do that. The step 3. should be protected against
61 * unsupported features first, because for now, most of them would be
62 * silently ignored.
63 *
64 * - No tool to "extract" an IOP² description of an IOP type for now. This
65 * feature could be useful for versioning and migration of IOP objects.
66 *
67 * - Sub-packages and multi-package loading.
68 */
